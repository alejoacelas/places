# Routing engines + map data + transit routing for 'trips' (drive + walk + transit + multimodal door-to-door)

> Intermediate research note — generated by the `derisk-trips-email-slack` workflow on 2026-06-29. This is raw input behind the plan docs, not the final word.

## Summary

A solo dev can build "Google Maps but better" for drive/walk/bike entirely on open-source with near-zero lock-in and only compute/storage cost — that layer is solved. Google's value is no longer routing math; it's the $200/month universal credit (removed March 1, 2025, now per-SKU free caps: Essentials 10k, Pro 5k, Enterprise 1k events/mo) plus its ToS, which forbid caching results (only place_id is storable indefinitely; lat/lng 30 days) and require any displayed route/Places result to sit on a Google basemap — directly contradicting "Maps but better" with a custom map. The real wall is data, not engines: metro transit needs per-agency GTFS + patchy GTFS-RT real-time feeds, and true long-distance/cross-border door-to-door (flights+rail+bus+ferry) has NO open-source engine and only closed APIs — Citymapper killed self-serve in June 2023 (Via/enterprise only) and Rome2Rio is now partner/sales-gated (owned by Omio, no public free tier). Recommended default: Valhalla (drive/walk/bike) + OpenTripPlanner 2 (transit) on OSM + GTFS, MapLibre GL rendering self-hosted Protomaps PMTiles, geocoding via Pelias/Photon — Google touched nowhere. The single dependency most likely to block is comprehensive + real-time transit/multimodal data, not any routing engine.

## Options

### Google Maps Platform (Routes API + Places + Maps JS)

*Managed APIs: Routes API (Compute Routes/Route Matrix, the successor to legacy Directions/Distance Matrix), Places API, and the Maps JS/SDK basemap. Pay-per-request with per-SKU free caps.*

_maturity: Mature, GA. Routes API is current; Directions/Distance Matrix are 'Legacy'. Pricing model changed 2025-03-01. · cost: Routes: $5/$10/$15 per 1k (Essentials/Pro/Enterprise) after small free caps. Places Details $5/1k (Essentials), Text/Nearby $32/1k (Pro). New accounts get a one-time $300 trial, not recurring._

**Pros**
- Best-in-class data quality, live traffic, global POI/transit coverage out of the box
- Zero ops; one API key, instant
- Routes API covers drive/walk/bike/transit in one call

**Cons**
- No universal $200/mo credit since 2025-03-01; replaced by thin per-SKU caps (Routes Essentials 10k/mo free then $5/1k; Pro $10/1k; Enterprise $15/1k). Places Text/Nearby Search are Pro at $32/1k
- ToS forbids caching/storing results except place_id (indefinite) and lat/lng (30 days) — you cannot pre-compute or cache to cut cost
- Displayed routes/Places MUST render on a Google basemap (or show Google logo+attribution if no map); you may NOT show Google content on/near a non-Google map — this kills 'custom map, better UX'
- Total vendor lock-in: ToS, data, and billing all Google
- Costs scale linearly with usage with no self-host escape hatch

### Self-hosted OSM routing: Valhalla / OSRM / GraphHopper

*Open-source engines that ingest an OpenStreetMap extract and answer route/matrix/isochrone over HTTP. Valhalla (C++, tiled, dynamic costing, has GTFS multimodal), OSRM (C++, contraction hierarchies, fastest, car/bike/walk only), GraphHopper (Java, flexible, has a GTFS 'pt' transit module).*

_maturity: Mature and production-grade for drive/walk/bike (OSRM/Valhalla/GraphHopper all >8 yrs). Valhalla transit is developing, not battle-hardened. · cost: Software free. One small VM (8-16GB RAM) per continent-sized extract; effectively $20-80/mo of hosting for personal scale._

**Pros**
- No per-request fee, no caching restriction, no display restriction — full control, results are yours
- OSM data is global and free (ODbL)
- Valhalla and GraphHopper-pt give multimodal/transit in the same engine; OSRM is fastest for pure car/walk
- Mature, widely deployed; Docker images exist

**Cons**
- You own ops: build pipelines, RAM/disk, OSM update cadence, monitoring
- OSRM has no transit and profile changes require full re-preprocessing; Valhalla GTFS transit is real but immature/buggy (e.g. transit isochrone tile-exclusion bug open Oct 2025); GraphHopper gates some matrix/optimization features behind its commercial license
- OSM has no live traffic — your ETAs are free-flow/historical unless you add a traffic source
- OSM data quality varies by region; ODbL share-alike obligations on derived geographic databases

### Self-hosted transit: OpenTripPlanner 2 / Navitia + GTFS data

*Dedicated multimodal transit routers. OTP2 (JVM, Java 25+) builds a graph from OSM + GTFS (+ Netex/SIRI), serves a GraphQL API, consumes GTFS-RT/SIRI for real-time. Navitia (open-source, also offered hosted) is a comparable French-origin engine.*

_maturity: OTP2 mature, dominant, actively maintained. Navitia mature but ops-heavy. · cost: Software free. A multi-GB metro/country graph needs 4-16GB+ RAM; Transitland Pro for feed discovery $200-250/mo (or free Mobility Database / Transitland hobbyist tier)._

**Pros**
- OTP2 is the gold standard for door-to-door transit incl. walk+transit, bike+transit, park-and-ride
- Reads open standards (GTFS, GTFS-RT, Netex, SIRI); real-time delays/cancellations supported
- Fully self-hostable, no per-query fee or display lock-in

**Cons**
- You must source and stitch a GTFS feed per agency/region — this is the real work; coverage is your responsibility
- GTFS-RT (live arrivals — the thing that beats static schedules) is fragmented, per-agency, and many agencies publish none
- OTP2 is RAM-hungry for large regions; JVM ops; REST API removed in 2025 (GraphQL only)
- Navitia is powerful but notoriously heavy to self-host

### Basemap/tiles: self-hosted Protomaps (PMTiles) vs Mapbox vs MapTiler

*The visual map layer + style, rendered client-side with MapLibre GL. Protomaps packages the whole planet basemap into one PMTiles file served from object storage/CDN; Mapbox and MapTiler are hosted tile/style vendors.*

_maturity: Protomaps + PMTiles + MapLibre all production-ready and widely adopted in 2024-2026. Mapbox/MapTiler mature. · cost: Protomaps: pennies-to-low-$ /mo on R2+Cloudflare. Mapbox: free to 50k loads then $5/1k. MapTiler: $0 (non-commercial) / $25 / $295._

**Pros**
- Protomaps = a single file on S3/R2 + Cloudflare, MapLibre frontend; near-zero cost and no per-load metering — the cost-killer for self-hosting
- MapLibre GL is the open, Mapbox-GL-compatible renderer (no Mapbox lock-in)
- Mapbox/MapTiler are turnkey with generous-ish free tiers if you don't want ops

**Cons**
- Protomaps you maintain (regenerate basemap on OSM updates, manage styles/glyphs/sprites)
- Mapbox free tier 50k map loads/mo then $5/1k; MapTiler free is non-commercial only, Flex $25/mo / Unlimited $295/mo
- Vendor tiles re-introduce metering and (Mapbox) historical license sensitivity

### Multimodal door-to-door aggregators: Rome2Rio / Citymapper

*Closed APIs that answer 'how do I get from town A to town B' across flights, rail, intercity bus, ferry, car — the long-haul/cross-border layer that no open engine covers.*

_maturity: Mature products, but as developer dependencies they are closing, not opening. · cost: Effectively inaccessible to a solo dev: enterprise/partner negotiation, undisclosed pricing._

**Pros**
- Only practical source for global long-distance multimodal itineraries
- Rome2Rio aggregates modes worldwide incl. obscure ferries/buses

**Cons**
- Citymapper self-serve SDKs/APIs discontinued 2023-06-23; now Via/enterprise ('Citymapper for Cities') only — closed to solo devs
- Rome2Rio is now partner/sales-gated (owned by Omio); the old free 100k/mo self-serve tier is effectively gone; no live prices via API
- No open-source equivalent exists; you cannot self-host this capability
- Enterprise contracts/NDAs unrealistic for a personal project

## Recommendation

**Pick:** Open-source self-host: Valhalla (drive/walk/bike) + OpenTripPlanner 2 (transit) on OSM + GTFS, rendered with MapLibre GL over self-hosted Protomaps PMTiles, geocoding via Pelias or Photon. Touch Google nowhere.

This delivers the drive/walk/bike core at production quality with zero per-request fees, zero caching ban, and zero 'must use our basemap' rule — the three things that make Google incompatible with the goal of a custom 'better' UX. Valhalla gives one engine for road modes (and an upgrade path to in-engine multimodal once its transit matures), while OTP2 is the proven transit router; both are free and yours. Protomaps+MapLibre removes the basemap-metering trap entirely. Choosing self-host also avoids Google's ToS, which legally forbids the exact product being described (custom map + cached/recombined results). The deliberate concession: skip true global long-distance door-to-door (the Rome2Rio domain) at v1 — scope to drive/walk/transit within regions where you have GTFS, because the only aggregators for the long-haul slice are closed to solo devs. Use OSRM instead of Valhalla only if you need maximum matrix throughput and don't need transit; use Mapbox/MapTiler only if you want to skip tile ops short-term.

## Key dependencies

- **GTFS + GTFS-RT transit data (per-agency feeds; Transitland / Mobility Database for discovery)** _(risk: high)_ — Whether transit routing exists at all in a given city, and whether it beats Google (needs live GTFS-RT, which most agencies don't publish or publish unreliably)
- **Long-distance multimodal aggregation (Rome2Rio / Citymapper — both closed to self-serve)** _(risk: high)_ — Cross-city/cross-border door-to-door itineraries; no open-source substitute, no realistic solo-dev API access
- **OpenStreetMap data (ODbL) coverage + freshness** _(risk: medium)_ — Road/path graph quality and the legality of redistributing derived data; no live traffic
- **OpenTripPlanner 2 runtime (JVM, RAM, GraphQL-only API)** _(risk: medium)_ — Transit graph build/serve; large regions need significant RAM and ops
- **Valhalla / OSRM / GraphHopper engine maturity (esp. Valhalla GTFS transit)** _(risk: medium)_ — Multimodal-in-one-engine reliability; Valhalla transit has open correctness bugs
- **Self-hosted basemap (Protomaps PMTiles + MapLibre GL)** _(risk: low)_ — The visible map and its cost curve; well-solved but you own regeneration/styles
- **Geocoding/POI search (Pelias / Photon / Nominatim) to avoid Google Places lock-in** _(risk: medium)_ — Search-a-place UX without Google Places' caching+display ToS; OSM POIs are thinner than Google's

## Derisking

### GTFS-RT real-time transit feeds

- **Risk:** Live arrivals/delays are what would make this 'better than Google,' but GTFS-RT is fragmented, per-agency, and frequently absent or stale; you can't manufacture it.
- **Evidence:** OTP2 supports GTFS-RT/SIRI updaters; Transitland covers 600k+ US stops; Mobility Database catalogs 6000+ feeds across 99 countries but RT coverage is far thinner than static.
- **Mitigation:** Start with cities that publish reliable GTFS-RT (use Transitland/Mobility Database to filter for RT feeds); fall back to static GTFS schedules elsewhere and label ETAs as scheduled; ingest RT via OTP2's SIRI/GTFS-RT updaters where available.
- **Fallback:** Static-schedule routing only in RT-less cities — still useful, just not 'better than Google' for live timing.

### Long-distance multimodal door-to-door (Rome2Rio/Citymapper)

- **Risk:** The cross-border flight+rail+bus+ferry layer has no open engine and both incumbent APIs are closed to solo devs (Citymapper self-serve dead since 2023-06-23; Rome2Rio partner/sales-gated, owned by Omio).
- **Evidence:** Citymapper 'SDKs and APIs come to an end' (2023-06-23); Rome2Rio docs path now 404s, access is partner-only with undisclosed pricing.
- **Mitigation:** Scope v1 to intra-region drive/walk/transit; treat long-haul as a later, separate problem; if pursued, stitch per-mode open data (OSM ferries, OpenRailRouting, intercity GTFS) yourself.
- **Fallback:** Link out to Rome2Rio/Google for long-haul legs rather than computing them in-house.

### Google Maps Platform ToS (if tempted to use Google for any layer)

- **Risk:** Caching/storing results is forbidden except place_id (indefinite) and lat/lng (30 days), and any displayed route/Places result must sit on a Google basemap — both rules directly forbid the intended custom-map, cached/recombined product.
- **Evidence:** Google Routes/Directions/Places policies: 'results displayed on a map must be shown on a Google Map'; 'do not pre-fetch, cache, or store... except place IDs.'
- **Mitigation:** Keep Google out of the stack entirely; use OSM/Valhalla/OTP2/Protomaps so results are yours to cache, recombine, and render on any map.
- **Fallback:** If Google is unavoidable for a feature (e.g. rich POI), isolate it to text-only/no-map flows and never cache beyond allowed windows.

### OTP2 / Valhalla operational footprint

- **Risk:** JVM RAM appetite and graph-build pipelines can overwhelm a solo dev as region size grows; Valhalla's GTFS transit has open correctness bugs.
- **Evidence:** OTP2 requires Java 25+ and is RAM-heavy; Valhalla transit isochrone tile-exclusion bug open as of Oct 2025.
- **Mitigation:** Containerize; start with one metro extract; schedule periodic OSM/GTFS rebuilds; use OTP2 for transit and Valhalla only for road modes until its transit stabilizes.
- **Fallback:** Use OSRM (lighter, faster) for road-only and skip in-Valhalla transit; cap covered regions to what one VM handles.

### OpenStreetMap data (ODbL, no live traffic)

- **Risk:** Variable regional quality, share-alike obligations on derived databases, and absence of real-time traffic mean drive ETAs are free-flow unless augmented.
- **Evidence:** OSM is ODbL-licensed; none of OSRM/Valhalla/GraphHopper ship live traffic.
- **Mitigation:** Provide proper OSM/ODbL attribution; for ETAs apply historical/speed-profile adjustments; consider an optional traffic source later.
- **Fallback:** Present ETAs as traffic-free estimates with a clear label; accept this as a known gap vs Google.

## Core interfaces

### Road routing (Valhalla HTTP)

```
POST /route { locations:[{lat,lon},{lat,lon}], costing:'auto'|'pedestrian'|'bicycle', costing_options:{...} } -> { trip:{ legs:[{ shape (encoded polyline), summary:{time,length}, maneuvers:[...] }] } }. OSRM equivalent: GET /route/v1/{profile}/{lon,lat;lon,lat}?steps=true&geometries=geojson.
```

### Transit routing (OTP2 GraphQL)

```
POST /otp/routers/default/index/graphql query plan(from:{lat,lon}, to:{lat,lon}, date, time, transportModes:[{mode:WALK},{mode:TRANSIT}]) { itineraries { duration startTime legs { mode route{shortName} from{name} to{name} legGeometry{points} realTime } } }
```

### Unified router facade (your normalization layer)

```
planTrip(origin, dest, modes:['DRIVE','WALK','TRANSIT'], departAt) -> Itinerary[] where Itinerary = { legs: Leg[], totalDuration, totalDistance }; Leg = { mode, polyline, fromStop?, toStop?, scheduled|realtime }. Fans out to Valhalla for road legs and OTP2 for transit, merges and ranks.
```

### Geocoding / place search (Pelias or Photon)

```
GET /v1/search?text=... -> GeoJSON FeatureCollection of {properties:{label,layer}, geometry:{coordinates:[lon,lat]}}. Reverse: GET /v1/reverse?point.lat=&point.lon=. Avoids Google Places caching/display ToS.
```

### Basemap rendering (MapLibre GL + PMTiles)

```
new maplibregl.Map({ style: protomapsStyleJSON }) where style sources = { protomaps: { type:'vector', url:'pmtiles://https://cdn/.../planet.pmtiles' } }; protocol registered via pmtiles.Protocol(). No per-load metering.
```

## Open questions

- Which cities/regions must v1 cover, and do those agencies publish GTFS-RT (live) or only static GTFS? This decides whether 'better than Google' for transit is even achievable there.
- Is in-region multimodal (drive+walk+transit) enough, or is global long-distance door-to-door (the Rome2Rio domain) a hard requirement? If the latter, the project needs a closed-API partnership that is likely out of reach for a solo dev.
- What does 'better' mean concretely vs Google — UX/customization, multimodal flexibility, privacy/no-tracking, offline, or routing quality? Open-source wins on the first set; Google still wins on live traffic and POI depth.
- Is live traffic-aware driving ETA in scope? OSM-based engines have none; would require an added (likely paid) traffic data source.
- Acceptable ops burden: is the dev willing to run OTP2/Valhalla on a VM with periodic graph rebuilds, or is a managed fallback (Mapbox/MapTiler for tiles, hosted Navitia for transit) preferred for speed-to-first-version?
- POI/search quality bar: are OSM-based geocoders (Pelias/Photon/Nominatim) sufficient, or is Google Places' richer POI data needed despite its caching+Google-basemap ToS lock-in?
