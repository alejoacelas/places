# trips — buying rail/travel tickets programmatically from the terminal (derisking the "integrate with Trainline and buy tickets" provocation)

> Intermediate research note — generated by the `derisk-trips-email-slack` workflow on 2026-06-29. This is raw input behind the plan docs, not the final word.

## Summary

Hard truth: no rail-ticket-selling API is self-serve for an individual in 2026. Every API that can actually take payment and issue a ticket — Trainline Partner Solutions, Omio B2B, Rail Europe, SilverRail, Amadeus Rail, Lyko, Distribusion — gates behind a signed B2B reseller/distribution contract, a registered business, and (for UK National Rail) RDG licensing plus an accredited Ticket Issuing System. The only credentials an individual can get free and instantly (Amadeus Self-Service, SNCF/Navitia, DB API Marketplace, UK Darwin via Rail Data Marketplace) are journey-planning, timetable, or price-lookup only and cannot sell anything. Trainline's sole individual-accessible channel is the Partnerize affiliate program — deep-link-and-commission, not transactional: you redirect the user to thetrainline.com to pay. So an individual cannot legitimately RESELL a ticket from a terminal command end-to-end; the only way to truly buy end-to-end from a terminal is to drive your own account's web checkout via browser automation (personal use, fragile, ToS gray area). The realistic MVP is: plan+price from open data, then affiliate deep-link to checkout, with an optional browser-automated buy on your own account behind a confirm prompt.

## Options

### Trainline Partner Solutions (TPS) Global API

*Trainline's B2B distribution API: 270+ rail/coach operators across 45 countries, full search/book/pay in 10 currencies plus aftersales (refunds, exchanges, journey changes). The real thing, but enterprise-only.*

_maturity: Production-grade, widely used (e.g. Amadeus Cytric UK rail content), but commercially inaccessible to an individual · cost: Undisclosed; commission/contract negotiated per partner_

**Pros**
- Broadest coverage in one API — UK National Rail, SNCF, DB, Renfe, Eurostar
- Trainline is merchant-of-record and owns fulfilment + aftersales
- Modern REST, cloud-hosted, security-accredited

**Cons**
- No self-serve, no public docs, no public pricing — 'get in touch' only
- ~12-week integration; built for funded travel sellers/carriers, not a solo dev
- Almost certainly requires a registered business and volume commitment; unlikely to contract an individual

### Trainline Affiliate via Partnerize

*The only Trainline channel open to individuals. Generate tracked deep-links to thetrainline.com; earn ~3% commission, 30-day cookie. No transaction happens in your code.*

_maturity: Mature, stable, individual-eligible · cost: Free to join; you earn commission rather than pay_

**Pros**
- Self-serve signup, no business strictly required, no joining fee
- Legitimate and ToS-compliant
- Pairs perfectly with a 'plan+price then open checkout' CLI

**Cons**
- NOT a booking API — user must finish payment on Trainline's site
- No programmatic price/availability feed; you only get links + reporting
- Commission is tiny (~3%); Partnerize expects a 'site/app' as the affiliate property

### Distribusion Reseller/Retailer API v2 (OSDM-based)

*Berlin aggregator; real search+book+pay+fulfilment across rail/coach/ferry. Public demo docs and an OSDM sandbox you can build against for free.*

_maturity: Production aggregator; sandbox is genuinely usable by a dev today · cost: Undisclosed contract/commission; sandbox free_

**Pros**
- Most accessible 'real booking' API — public demo + sandbox to prototype now
- OSDM-standard, growing rail coverage in EU/NA
- Handles fulfilment (e-ticket/QR) and acts as the distribution layer

**Cons**
- Production access needs a signed retailer contract; aimed at OTAs/TMCs/registered businesses
- Min commercial terms undisclosed; unlikely to onboard a hobbyist
- You may become a sub-retailer carrying consumer-law/Seller-of-Travel obligations

### Omio B2B Booking API / Rail Europe API / SilverRail (SilverCore)

*Three aggregator/distribution APIs that genuinely sell tickets across many operators via one connection.*

_maturity: Production-grade, enterprise distribution · cost: Undisclosed; negotiated commercial agreements_

**Pros**
- Full booking + payment + multi-operator coverage
- Rail Europe is an official agent across 24 countries; SilverRail powers GDS/TMC rail
- Single integration instead of per-operator deals

**Cons**
- All 'contact sales' / B2B-contract only; no self-serve, no public pricing
- Built for travel platforms/agencies, not individuals
- Still need per-operator approval behind the unified API

### Amadeus Rail API

*Amadeus rail search/shopping/booking/ticketing, 90+ operators — but sits under the ENTERPRISE catalog (Amadeus Web Services Rail), not the free Self-Service tier.*

_maturity: Mature GDS, but gated · cost: Enterprise contract; Self-Service free tier does not include rail booking_

**Pros**
- Strong content + GDS-grade ticketing
- Good docs and developer tooling
- Test environment exists

**Cons**
- Rail is enterprise-only — requires an Amadeus commercial agreement; Self-Service covers flights/hotels/cars, NOT rail booking
- Heavy onboarding, IATA/agency-style relationship expected
- Not realistic for a solo individual

### Lyko / Travelfusion tfRail (third-party rail APIs)

*Smaller aggregators exposing DB, SNCF, Renfe etc. with full search+booking via one API (Lyko) or XML (Travelfusion).*

_maturity: Operating commercial APIs; smaller scale · cost: Undisclosed; commission share_

**Pros**
- Real booking incl. DB e-tickets
- Single API across several national operators
- Commission-sharing model

**Cons**
- Sales-contact only, business-oriented, no self-serve or public pricing
- Smaller/less-proven than Trainline/Amadeus
- Coverage and reliability need verification

### National operator + open-data APIs (SNCF/Navitia, DB API Marketplace, UK Darwin/Rail Data Marketplace, Renfe Open Data)

*The free, instantly self-serve APIs an individual CAN get — but they are journey-planning, timetable, realtime, and price-LOOKUP only. None sell tickets.*

_maturity: Mature and reliable for planning/pricing · cost: Free (quota-limited)_

**Pros**
- Free, self-serve API keys today, no contract
- Excellent for planning, live times, fares discovery (SNCF Navitia; DB timetables; UK Darwin LDBWS; season-ticket/fare price products)
- Stable, documented, legitimate

**Cons**
- Cannot book or pay — info-only by design
- Booking on SNCF Connect / bahn.de / renfe.com is web/app only, no public purchase API
- Fragmented: per-country keys and data models

### Browser automation of your own account (Playwright / claude-in-chrome) — the only individual end-to-end BUY

*Drive the real thetrainline.com / sncf-connect.com / bahn.de checkout headfully with your own saved card, from a terminal command, for your own trips (personal use, not resale).*

_maturity: Technically feasible today; brittle, unsupported, needs a spike · cost: Free; cost is maintenance + breakage_

**Pros**
- The ONLY way an individual can truly buy a ticket from a terminal end-to-end in 2026
- No contract, works against the live consumer site
- Fits 'plan in terminal, then complete the purchase' UX

**Cons**
- Fragile: UI changes, Cloudflare/bot detection, CAPTCHA, 3-D Secure step-up
- ToS gray area — most sites prohibit automated access (personal use mitigates but doesn't license it)
- Fulfilment may be app-locked (mobile-only Aztec/QR tickets unusable outside their app)
- You handle your own card entry; high-friction to make robust

## Recommendation

**Pick:** Phased MVP: (0) plan+price in the terminal from free open-data APIs; (1) generate a Partnerize-affiliate deep-link to the Trainline/SNCF/Omio checkout and `open` it; (2) optional 'buy' that browser-automates YOUR OWN account checkout behind an explicit confirm-before-pay prompt. Do NOT pursue a reseller API for the MVP.

This is the only path an individual can actually ship in 2026. Phases 0-1 are fully legitimate, contract-free, and deliver 'one command to a ready-to-pay checkout' — the realistic reading of 'buy tickets from the terminal.' Phase 2 is the only genuine end-to-end buy available to an individual, but it is brittle and ToS-gray, so it stays optional, personal-use, and gated behind confirmation. Real resale (Distribusion/Omio/TPS) only makes sense if 'trips' becomes an incorporated business — defer it; prototype against the Distribusion sandbox if you want to prove the booking flow without going live.

## Key dependencies

- **Trainline Partner Solutions Global API (booking)** _(risk: high)_ — Signed B2B contract, registered business, ~12-week integration, undisclosed volume/commission terms; no self-serve
- **Trainline Affiliate (Partnerize) deep-links** _(risk: low)_ — Self-serve signup; Partnerize expects a website/app property; deep-link only, no transaction or price feed
- **Distribusion Reseller API (real booking, OSDM)** _(risk: high)_ — Free sandbox now; production needs signed retailer contract + registered business + likely Seller-of-Travel/consumer obligations
- **Amadeus Rail / Omio B2B / Rail Europe / SilverRail / Lyko** _(risk: high)_ — All enterprise sales-contract only; no self-serve; rail not in Amadeus free Self-Service tier
- **Open-data planning APIs (SNCF Navitia, DB Marketplace, UK Darwin/RDM, Renfe Open Data)** _(risk: low)_ — Free self-serve API keys, monthly quotas; info/price-lookup only, cannot book
- **Browser automation of own consumer checkout** _(risk: high)_ — No contract; fights bot-detection/CAPTCHA/3DS; ToS prohibits automated access; mobile-only ticket fulfilment may be unusable outside operator app
- **UK National Rail retailing (if reselling UK tickets)** _(risk: high)_ — RDG third-party retailer license + accredited Ticket Issuing System (RSP) + LENNON settlement connection; in practice sub-retail via Trainline or Evolvi
- **Payment & PCI / merchant-of-record** _(risk: high)_ — If you ever take card payment yourself: PCI-DSS SAQ + chargeback/fraud liability. Avoided by letting the aggregator/Trainline be merchant-of-record

## Derisking

### Trainline booking for an individual

- **Risk:** There is NO individual-accessible Trainline booking API. TPS is enterprise-only; the affiliate channel cannot transact. The provocation 'buy tickets via Trainline API' is not achievable as a solo dev.
- **Evidence:** TPS Global API page is 'get in touch' / ~12-week integration with no public pricing; Trainline publishes no public developer booking API; only the Partnerize affiliate is open to individuals.
- **Mitigation:** Reframe to 'plan+price then deep-link to Trainline checkout' via the Partnerize affiliate program (individual-eligible) and finish payment on Trainline's site.
- **Fallback:** Browser-automate your own Trainline account checkout for personal trips.

### Any 'real booking' aggregator API (Distribusion/Omio/RailEurope/SilverRail/Amadeus/Lyko)

- **Risk:** All require a signed B2B contract and registered business; none are self-serve. An individual cannot get production booking access.
- **Evidence:** Distribusion has public demo docs + sandbox but production is retailer-contract; Omio/RailEurope/SilverRail/Lyko are all 'contact sales'; Amadeus rail sits in the Enterprise catalog, not free Self-Service.
- **Mitigation:** Prototype the full book flow free against the Distribusion demo/OSDM sandbox to prove feasibility without going live; only sign a contract if 'trips' incorporates.
- **Fallback:** Stay at deep-link + affiliate; never take payment yourself.

### End-to-end purchase from a terminal command

- **Risk:** The only self-serve APIs (SNCF/Navitia, DB Marketplace, UK Darwin, Renfe Open Data, Amadeus Self-Service) are info/price-only and cannot complete a purchase.
- **Evidence:** SNCF API is Navitia journey-planning/open-data; DB API Marketplace is timetables/stations/realtime with no public booking; UK Darwin/LDBWS is information-only; Renfe is open-data only.
- **Mitigation:** Use them for plan+price; complete payment via affiliate deep-link in the browser, or via browser automation of your own account for personal use.
- **Fallback:** Accept 'one command to a pre-filled, ready-to-pay checkout' as the shippable definition of done.

### Browser-automated personal purchase

- **Risk:** Brittle (UI churn, Cloudflare/bot detection, CAPTCHA, 3-D Secure), ToS prohibits automated access, and many EU/UK tickets are mobile-app-locked (Aztec/QR only in operator app) so the ticket may be unusable from a terminal.
- **Evidence:** Reverse-engineered Trainline wrappers (e.g. tducret/trainline-python) are unmaintained since ~2020 and effectively broken, illustrating how fragile and ToS-adverse this surface is.
- **Mitigation:** Run a 1-day spike against thetrainline.com and sncf-connect.com to test bot-detection survivability and whether a print-at-home/PDF (PDF417) fulfilment option exists for the routes you care about. Keep it headful, personal-account, confirm-before-pay.
- **Fallback:** If automation/fulfilment fails, the deep-link 'open checkout' flow still ships and is robust.

### UK National Rail resale + payment compliance

- **Risk:** Selling National Rail tickets requires an RDG license, an accredited TIS, and settlement-system connection; taking card payment yourself triggers PCI-DSS and fraud/chargeback liability.
- **Evidence:** RDG requires a third-party retailer license; every system selling inter-available tickets must pass RSP Accreditation and use an approved Ticket Issuing System.
- **Mitigation:** Never become the retailer/merchant-of-record for an MVP; rely on Trainline/aggregator as merchant-of-record so you inherit their accreditation and PCI scope.
- **Fallback:** Sub-retail through an already-accredited partner (Trainline, Evolvi) only if a business case emerges.

## Core interfaces

### RailProvider (capability-tiered abstraction)

```
interface RailProvider { search(q: JourneyQuery): Promise<Itinerary[]>; price?(it: Itinerary): Promise<Fare[]>; checkoutUrl?(it: Itinerary): URL;  // affiliate-tagged deep link\n  book?(it: Itinerary, pay: PaymentRef): Promise<Booking>;  // contract-gated, not in MVP\n  capabilities: { plan: bool; price: bool; deeplink: bool; book: bool } }\n// Implementations: SncfNavitia(plan,price?), DbMarketplace(plan), UkDarwin(plan,price), TrainlineAffiliate(deeplink), DistribusionSandbox(book) — only sandbox sets book:true.
```

### CLI surface

```
trips plan "Paris -> Lyon" --date 2026-07-10 --time 09:00   # itineraries from open data\ntrips price <itineraryId>                                    # fares via lookup/scrape\ntrips checkout <itineraryId>                                 # open() affiliate deep-link to Trainline/SNCF\ntrips buy <itineraryId> --account me --confirm              # OPTIONAL: browser-automate own checkout, pay after explicit confirm
```

### CheckoutDeepLink

```
function checkoutUrl(it: Itinerary): URL {\n  // Partnerize-wrapped link to thetrainline.com prefilled search/results\n  // e.g. https://prf.hn/click/<camref>/destination/<encoded thetrainline.com results url>\n  return wrapAffiliate(buildTrainlineSearchUrl(it), PARTNERIZE_CAMREF);\n}
```

### AutomatedBuy (Phase 2, personal use)

```
async function buy(it, account, opts){\n  const page = await openHeadful('https://www.thetrainline.com');\n  await login(page, account);          // user's own saved creds\n  await fillJourney(page, it);\n  const summary = await readPriceAndTerms(page);\n  if(!await confirm(summary)) abort(); // confirm-before-pay (irreversible gate)\n  await completePayment(page);         // saved card + 3DS handled in-browser\n  return await captureTicket(page);    // PDF/print-at-home if offered; else flag app-locked\n}
```

## Open questions

- Will headful browser automation survive Cloudflare/bot-detection and 3-D Secure on thetrainline.com and sncf-connect.com in 2026? Needs a 1-day spike before committing to Phase 2.
- Fulfilment blocker: for the routes you care about, is a print-at-home/PDF (PDF417) ticket available, or are tickets mobile-app-locked (Aztec/QR in the operator app)? If app-locked, a terminal purchase yields a ticket you can't use outside their app.
- Will Partnerize approve a personal CLI/tool as a valid affiliate property, and are deep-links allowed to prefill search/results so the UX is genuinely one-command-to-checkout?
- Distribusion's real onboarding bar: would they contract a solo dev with a one-person LLC, and what are the minimum commercial/volume terms and Seller-of-Travel/consumer obligations?
- If you ever want true API booking, is sub-retailing through Trainline/Evolvi (inheriting their RDG accreditation + merchant-of-record + PCI scope) cheaper/faster than any direct distributor contract?
- Scope question for 'trips': is the goal personal travel (favor open-data planning + own-account automation) or a product that sells to others (which forces incorporation + a reseller contract)? The architecture differs sharply.
