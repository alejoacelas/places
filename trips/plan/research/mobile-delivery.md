# trips — "can we carry it on my cellphone?" (delivering a CLI-origin travel tool to an iPhone)

> Intermediate research note — generated by the `derisk-trips-email-slack` workflow on 2026-06-29. This is raw input behind the plan docs, not the final word.

## Summary

The trips tool is a CLI-origin travel planner/ticket-buyer (per friends/places/trips/README.md, still planning-stage) that wants a phone frontend and to stay drivable by Claude Code. The decisive architectural move is to keep a small self-hosted HTTP API wrapping the CLI as the single backend spine; every phone option then becomes a thin, swappable frontend, and Claude Code keeps driving the same CLI/API regardless. Among the six paths, a Telegram bot is the lowest day-to-day friction "in my pocket" surface: the client is already installed, push is free and arrives on the lock screen via Telegram's own infrastructure (no APNs, no Apple Developer account, no PWA install ritual), auth is a one-line user-ID allowlist, and with long-polling the home server needs no public endpoint, port-forward, or TLS cert at all. The real weakness is that chat is a poor surface for the visual route-comparison and maps that are the product's actual promise ("Google Maps but better"), so Telegram should own commands + notifications while a thin PWA is added later for visual planning. The genuine project-killer is unrelated to phone delivery: it is the ticket-buying integrations (Trainline et al.), which the README itself flags as the hard half.

## Options

### PWA (installable web app) + self-hosted API

*An add-to-home-screen web app served by a small self-hosted HTTP API (e.g. FastAPI) that wraps the trips CLI; map/route UI renders in WebKit.*

_maturity: Web stack mature; iOS install+push is the perennial weak link · cost: Free (self-host + Cloudflare Tunnel free tier)_

**Pros**
- Rich visual UI (maps, side-by-side route comparison) — the only option that directly serves the 'Google Maps but better' promise
- One codebase, cross-platform
- Web push has worked on iOS since 16.4; iOS 26 opens home-screen sites as web apps by default and Safari 18.4 added service-worker-free Declarative Web Push
- Claude Code drives the exact same HTTP API

**Cons**
- iOS push ONLY works after a manual Share -> Add to Home Screen install plus a notification-permission tap; lost on every reinstall and far lower reach than native
- No silent push, no background wake
- Needs a public HTTPS endpoint (Cloudflare Tunnel) for both serving and push
- Offline only if you hand-build service-worker caching

### Chat bot frontend — Telegram (vs WhatsApp)

*A bot you DM commands to that proxies to the trips CLI/API and replies in chat; Telegram or WhatsApp as the channel.*

_maturity: Very mature — Telegram Bot API free and stable since 2015 · cost: Telegram free; WhatsApp ~$0.008-0.02/message + BSP platform fees_

**Pros**
- Zero install friction — Telegram is already on the phone
- FREE push via Telegram's own infra: lock-screen notifications with no APNs, no PWA install, no Apple Developer account
- With long-polling the home server needs NO public endpoint, port-forward, or TLS cert — it reaches out to Telegram
- Trivial auth: allowlist your own Telegram user ID
- Bot API free at any message volume
- Claude Code can hit the same backend or even post through the bot

**Cons**
- Chat is a weak surface for maps / visual route comparison — the product's core value
- WhatsApp variant is far heavier: requires a Business Solution Provider, business verification, and per-message billing (~$0.008-0.02/msg)
- No true offline; depends on connectivity (acceptable for a booking tool)
- Long-polling holds one connection per token (concurrent pollers get 409)

### iOS Shortcuts -> self-hosted HTTP endpoint

*A Shortcut using 'Get Contents of URL' to POST to your API, triggered from a home-screen icon, Siri, Share Sheet, Back Tap, or an automation.*

_maturity: Mature but quirky; automation auto-run behavior shifts release to release · cost: Free_

**Pros**
- Near-zero build cost
- Native feel: Siri, home-screen tile, share-sheet, Back Tap
- Best Claude-Code parity — identical HTTP API
- Excellent for one-shot actions ('next train to X', 'buy it')

**Cons**
- No real UI beyond text/list/menu prompts — useless for visual planning
- Cannot RECEIVE push; can only display a notification while the shortcut is running
- Time/location automations still frequently force 'Ask Before Running' (only Wi-Fi/Bluetooth run silently on iOS 26; users report 18.2 confirmation regressions)
- Needs a reachable HTTPS endpoint
- Fiddly for multi-step flows

### Tailscale + SSH to a home server

*Phone joins your WireGuard tailnet and SSHes (Blink/Termius) into the Mac/home server to run the trips CLI directly, ideally inside tmux.*

_maturity: Very mature (WireGuard-based) · cost: Free_

**Pros**
- Lowest BUILD friction — runs your existing CLI unchanged, no API to write
- Claude Code already drives that exact CLI
- Tailscale free Personal plan: 6 users, unlimited devices (expanded April 2026)
- No public ports; Tailscale SSH manages auth
- mosh + tmux survive network changes

**Cons**
- Highest DAY-TO-DAY friction on iOS — typing CLI on a phone keyboard
- No push at all
- Home server must stay up and reachable
- It is 'remote into my desk', not really 'in my pocket'

### Termux (Android-only — the iOS gap)

*A full local Linux terminal + package manager on the phone. Android ONLY; no iOS build exists or can exist under App Store/JIT rules.*

_maturity: Mature on Android; not applicable on iOS · cost: Free_

**Pros**
- On Android only: run the CLI locally on the phone, fully offline, no server needed

**Cons**
- IRRELEVANT here — the user is on iPhone
- iOS substitutes are all compromised: iSH is slow x86 Alpine emulation with networking limits and no JIT; a-Shell is a sandboxed local WASM/Unix toolbox with no general package ecosystem; Blink/Termius are SSH clients that collapse into the Tailscale+SSH option
- None reproduce Termux's local package-manager experience on iOS

### Thin native / Expo app

*A React Native/Expo app calling the self-hosted API, with real native UI and APNs push.*

_maturity: Mature toolchain (Expo/EAS), heavy process · cost: $99/yr (Apple Developer Program) or free with 7-day re-sign pain_

**Pros**
- Highest UX ceiling: maps, native push, offline storage, background tasks
- Proper push via APNs/Expo
- TestFlight distribution path

**Cons**
- Highest build and maintenance burden
- Needs Apple Developer Program at $99/yr, OR the free-account treadmill (App IDs expire every 7 days, max 3 devices, weekly re-sign)
- Store/TestFlight overhead
- Still requires the same backend API anyway
- Overkill for a solo personal tool at planning stage

## Recommendation

**Pick:** Telegram bot on top of a small self-hosted HTTP API, with the backend talking to Telegram via long-polling (no public endpoint required). Add a thin PWA later for the map/visual layer.

It is the lowest-friction way to put trips in your pocket today: nothing to install (Telegram is already there), free lock-screen push with no APNs/Apple Developer account/PWA-install ritual, one-line user-ID auth, and — because long-polling has the home server dial out — zero public exposure, no port-forwarding, no TLS cert. The self-hosted API spine means Claude Code keeps driving the identical backend (or posts via the bot), and a PWA/Shortcuts/native frontend can be added on top without rework. The main risk is that chat is a weak surface for the visual route comparison that is the product's core promise, so scope Telegram to commands + notifications and plan a PWA web view for visual planning. Note this risk is downstream of the project's actual hard problem — the ticket-buying integrations — which dominates feasibility regardless of phone delivery.

## Key dependencies

- **Telegram Bot API (free, long-polling)** _(risk: low)_ — The push + command channel and the whole recommendation
- **Self-hosted HTTP API wrapping the trips CLI** _(risk: medium)_ — Every frontend (PWA, bot, Shortcuts, native) and Claude-Code parity
- **Home server uptime + residential network** _(risk: medium)_ — Availability of the bot/API when traveling
- **Tailscale free Personal plan** _(risk: low)_ — Secure remote reach to the home server and the SSH fallback
- **Cloudflare Tunnel (only if you pick webhook, PWA, or Shortcuts)** _(risk: low)_ — Public HTTPS reachability without opening ports
- **Ticket-purchase integrations (Trainline et al.)** _(risk: high)_ — The product's 'buy from terminal' promise — the real feasibility blocker
- **Apple Developer Program ($99/yr) — only for the native/Expo path** _(risk: medium)_ — Native build + APNs push + non-expiring install

## Derisking

### Home server uptime + residential network behind the long-polling bot

- **Risk:** While traveling the bot/API silently dies if the home server reboots, the ISP drops, or polling stalls — exactly when you need it most
- **Evidence:** Telegram rejects concurrent getUpdates with 409; community notes (OpenClawAI 2026) on polling resilience and dropped messages on self-hosted setups
- **Mitigation:** Run the bot as a supervised service (launchd/systemd or Docker restart=always) with a heartbeat; consider a small always-on box (Pi/mini-PC) rather than the laptop; add a watchdog that re-establishes getUpdates
- **Fallback:** Move the API to a cheap always-on VPS, or front it with Cloudflare Tunnel; Tailscale+SSH remains a manual escape hatch

### Telegram as the primary UI surface

- **Risk:** Chat cannot render maps or side-by-side multimodal route comparison — the core 'Google Maps but better' value is lost if you stop at the bot
- **Evidence:** friends/places/trips/README.md frames the product as visual route comparison + ticket buying
- **Mitigation:** Scope Telegram to commands + notifications (departures, delays, 'buy it'); design the shared API so a PWA map view can be added with no backend change; have the bot deep-link into the PWA for visual results
- **Fallback:** Promote the PWA to the primary planning surface and keep Telegram only for push/quick actions

### iOS push delivery

- **Risk:** If you ever switch from Telegram to PWA/Shortcuts for notifications, iOS push gets fragile — PWA push needs manual Add-to-Home-Screen + permission and breaks on reinstall; Shortcuts cannot receive push at all
- **Evidence:** Apple's web push requires PWA installed to Home Screen; Shortcuts can only show a notification while running
- **Mitigation:** Keep push on Telegram, which uses its own delivery and needs no Apple entitlements
- **Fallback:** Native/Expo app with APNs if rich/reliable push ever becomes a hard requirement (accept the $99/yr + maintenance)

### Auth on the self-hosted API

- **Risk:** A publicly reachable endpoint wrapping a tool that can spend money (buy tickets) is a real exposure if left open
- **Evidence:** Long-polling avoids any public endpoint entirely; Tailscale SSH/ACLs and Cloudflare Access are standard auth fronts
- **Mitigation:** Bot path: allowlist your Telegram user ID and keep a bearer token between bot and API. API path: bind to Tailscale-only, or Cloudflare Access in front of the Tunnel
- **Fallback:** Keep the purchase action behind an explicit confirm step and a separate scoped secret; never expose buy endpoints on the open internet

### Termux as a phone-local option

- **Risk:** Planning around Termux wastes effort — it does not and cannot exist on iOS
- **Evidence:** Termux is Android-only; iOS substitutes (iSH/a-Shell) are sandboxed/emulated and Blink/Termius are just SSH clients
- **Mitigation:** Treat iOS as server-backed by default; if true offline-local execution is ever needed, that is an Android-only or native-app conversation
- **Fallback:** a-Shell for trivial local scripts, or accept connectivity dependence (a travel-booking tool needs the network anyway)

## Core interfaces

### trips HTTP API (the shared spine)

```
Small FastAPI/Express service wrapping the CLI. e.g. POST /route {from,to,modes,prefs} -> ranked options; POST /book {option_id} -> {needs_confirm, order}; POST /book/confirm {order_id}; GET /trip/{id}. Auth: bearer token. Same process the CLI already calls, so Claude Code hits it directly.
```

### Telegram bot (long-poll loop)

```
while true: updates = getUpdates(offset, timeout=50); for u in updates: if u.from.id != ALLOWED_ID: skip; dispatch command -> call trips API -> reply. Runs on home server, dials out to api.telegram.org; no inbound port. Supervised with restart=always.
```

### Push / notification path

```
Server-side events (gate change, delay, 'ticket bought') -> sendMessage(ALLOWED_ID, text, inline_keyboard). Delivered to lock screen by Telegram. No APNs, no entitlement.
```

### iOS Shortcut (optional glue)

```
Get Contents of URL: POST https://<tunnel>/route, Authorization: Bearer <token>, body JSON; parse reply into a Show Result / Choose from Menu. Trigger via Siri or home-screen tile.
```

### Claude Code automation entry

```
Claude Code runs the same CLI locally OR curls the trips API; for proactive messages it can call the bot's sendMessage. One backend, every driver.
```

## Open questions

- Does the user actually need offline planning on the phone, or is always-connected fine for a booking tool? If offline matters, only Termux (Android) or a native app with local cache satisfies it — and the user is on iOS.
- Visual layer: is a PWA map view enough, or is native-grade maps/interaction wanted? This decides whether the $99/yr native path is ever justified.
- Should Claude Code automations be able to reach OUT to the phone (proactive nudges via the bot), or only share the backend? The bot makes proactive push trivial.
- Where does the always-on backend live — the MacBook, a Pi/mini-PC, or a cheap VPS? This dominates the uptime risk while traveling.
- The dominant unknown is upstream of phone delivery: do Trainline/rail providers permit programmatic ticket purchase under their ToS/anti-bot terms? The README already flags this as the hard half and it gates the whole 'buy from terminal' premise.
